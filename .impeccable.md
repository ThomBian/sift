# Sift — Design Standard

> A living document. Every design decision in this product should trace back to a principle here.

---

## Design Context

### Users

Productive people — makers, developers, operators — who experience cognitive drag from bloated task tools. They use Sift primarily on desktop and in-browser during deep work sessions, and on mobile when capturing on the go. They don't want to manage a tool; they want the tool to disappear so they can do the actual work.

The emotional job Sift does: **make the user feel in control and unencumbered**. Not "productive" in the hustle sense — present, focused, and mentally free.

### Brand Personality

**Sharp. Fast. Opinionated.**

- **Sharp**: Precise, surgical, no wasted space. Every pixel earns its place.
- **Fast**: Responses feel instantaneous. Motion confirms, never celebrates.
- **Opinionated**: The tool makes decisions for the user. No settings sprawl. No "many ways to do this." One right way, clearly expressed.

The closest cultural reference is Linear — not because we copy it, but because Linear has the same relationship with its users: deeply respectful of their time, keyboard-native, visually confident. Sift should feel like Linear if Linear were built around the philosophy of mental unloading rather than project management.

**Anti-references**: JIRA (bureaucratic overhead), Confluence (infinite document bucket), Notion (infinite flexibility = no opinions). The opposite of Sift is a tool that makes you feel like you need to organize before you can work.

### Core Metaphor

Sift is a **conveyor belt**, not a bucket. Tasks flow through it with intention. The UI must reinforce this: nothing stagnates, everything resolves. The interface should feel like a system that processes, not a container that stores.

---

## Design Principles

### 1. The Conveyor Belt Principle

The UI reinforces forward motion. Tasks move through states (inbox → working → done). Nothing should look like it's piling up. Focused, active, and late states should each feel distinct — not through decoration, but through precise typographic and color signals.

### 2. Zero Tolerance for Decoration

Every visual element either communicates information or reduces cognitive load. If you're adding something for aesthetic reasons alone, remove it. No decorative cards, no gradient fills, no flourishes that don't earn their place.

### 3. Speed as Respect

The interface feels instantaneous. 150ms max on all transitions. Optimistic updates everywhere. Motion confirms user intent — it never lingers to admire itself. A user who types fast should never feel the UI is catching up.

### 4. Opinionated Defaults

Sift makes decisions for the user. The UI should reflect this — confident, direct, with no "which way do you prefer?" moments. One keyboard shortcut per action. One visual treatment per state. No configuration theater.

### 5. Precision Engineering

Sharp edges, tight spacing, surgical hierarchy. The tool should feel like a scalpel. Zero border-radius everywhere — this is non-negotiable and communicates intentionality. Soft corners belong to consumer apps; Sift is a professional instrument.

## Color System

### Philosophy

Colors in Sift are signals, not decoration. The accent (#FF4F00) is rare and meaningful — it marks focus, urgency, and active state. Neutrals are slightly warm-tinted toward the accent hue for subconscious cohesion.

### Current Palette

| Token       | Value     | Role                          |
| ----------- | --------- | ----------------------------- |
| `bg`        | `#FFFFFF` | Page background               |
| `surface`   | `#FAFAFA` | Sidebar, header surfaces      |
| `surface-2` | `#F0F0F0` | Hover states                  |
| `border`    | `#E2E2E2` | Retina borders (use at 0.5px) |
| `border-2`  | `#CCCCCC` | Stronger borders, checkbox    |
| `text`      | `#111111` | Primary text                  |
| `muted`     | `#888888` | Secondary metadata            |
| `dim`       | `#BBBBBB` | Disabled, placeholder         |
| `accent`    | `#FF4F00` | Focus, active, key actions    |
| `red`       | `#E60000` | Late tasks, destructive       |
| `green`     | `#16A34A` | Done state                    |

### Refinements to Consider

**Tint neutrals toward orange.** The white background should carry a barely-perceptible warmth — `oklch(99.5% 0.004 50)` instead of pure `#FFFFFF`. Pure white exists nowhere in nature; a slight tint creates subconscious cohesion with the accent.

```css
/* Proposed OKLCH equivalents — perceptually uniform */
--color-bg: oklch(99.5% 0.004 50); /* barely warm white */
--color-surface: oklch(98% 0.005 50); /* sidebar warmth */
--color-surface-2: oklch(95% 0.006 50); /* hover warmth */
--color-border: oklch(90% 0.005 50); /* retina border */
--color-text: oklch(15% 0.008 50); /* near-black, warm */
--color-muted: oklch(58% 0.006 50); /* warm gray */
--color-accent: oklch(61% 0.21 42); /* #FF4F00 in OKLCH */
```

**The accent is the product's signature.** Never dilute it by applying it to non-critical elements. It should appear only on: focused task rows, active nav items, key action buttons, the sync indicator when active, selection text.

### Dark Mode (Future Consideration)

Given the "desktop during deep work" context, a dark theme is worth exploring. Users like Linear users often prefer dark for sustained desktop sessions. If pursued:

- Base: `oklch(12% 0.01 50)` — very dark warm near-black
- Surface: `oklch(16% 0.012 50)` — slightly lighter for sidebar/header
- Accent stays: `#FF4F00` glows even harder on dark

## Typography

### Current System

| Role                   | Font           | Size             | Weight  | Tracking               |
| ---------------------- | -------------- | ---------------- | ------- | ---------------------- |
| Task title             | Geist          | `text-sm` (14px) | 500     | `-0.02em`              |
| Section header         | JetBrains Mono | `10px`           | 400     | `0.2em` uppercase      |
| Metadata (date, count) | JetBrains Mono | `10-12px`        | 400     | normal/tabular         |
| Nav labels             | JetBrains Mono | `11px`           | 400/600 | `0.1-0.35em` uppercase |

### Principles

**Use fewer sizes with more contrast.** The current system has too many sizes that are too close together (10px, 11px, 12px, 14px). Collapse to a cleaner scale with more ratio between steps.

Recommended fixed scale for app UI:

```
10px — metadata, section labels (JetBrains Mono)
13px — secondary task info, project names
15px — task titles (primary reading size)
20px — view headings (if needed)
```

**Line-height and tracking.** Tight tracking (`-0.02em`) on titles is correct for Geist at weight 500. Keep it. For monospace metadata, the default tracking is fine — don't add extra `letter-spacing` to already-spaced mono fonts.

**Hierarchy through weight and color, not size alone.** Task title in `#111111` weight 500, metadata in `#888888` weight 400 — this two-tier signal is already working. Reinforce it, don't muddle it.

### Font Audit Note

Geist is not in the banned reflex list and works for a developer-adjacent productivity tool. If ever revisiting, look at:

- **Recia** or **Editorial New** for a display layer (section headings)
- **Martian Mono** as an alternative to JetBrains for metadata — slightly more distinctive

## Spacing

### Scale (4pt base)

```
4px  — micro: gap within a chip, icon padding
8px  — small: gap between icon and label
12px — compact: internal row padding, chip gaps
16px — base: section padding
24px — medium: section separation
32px — large: view-level padding
48px — xl: empty state vertical centering
```

### Row density

Task rows at `36px` are tight and correct for a power user tool. Do not increase this. If anything, 32px is worth exploring for the highest-density feel.

### Sidebar width

`192px` (w-48) is correct. Do not widen. The sidebar should feel like a utility rail, not a navigation panel.

## Motion

### Philosophy

Sift's motion is **confirmatory, not performative**. Animations tell the user "I heard you" — they don't put on a show. Every animation should feel like it has a clear physical analog.

### Timing

```
100ms  — micro feedback (button press, checkbox check)
150ms  — standard (palette open/close, task enter)
180ms  — entrance with spring (task appears)
0ms    — state changes that don't need emphasis (color switches)
```

Never exceed 200ms for task-related animations. The user is moving fast; the UI must keep up.

### Easing

```
Entry:  cubic-bezier(0.34, 1.56, 0.64, 1)  — slight spring, task appears to snap into place
Exit:   cubic-bezier(0.23, 1, 0.32, 1)     — smooth ease-out, no overshoot
Active: cubic-bezier(0.23, 1, 0.32, 1)     — press states, color transitions
```

The spring easing for entry (slight overshoot) is correct and distinctive. Keep it.

### Key Animations

**Task enter**: Slide in from left 6px + fade. Staggered by index (max 150ms delay). Communicates "new items are arriving."

**Task exit**: Slide right 10px + fade. Quick (150ms). Communicates "resolved and gone."

**Palette open/close**: Scale + translate + fade. Feels like a panel dropping from the action bar.

**Modal enter/exit**: Scale + translate Y. Slightly slower than palette (150/120ms) — modals carry more weight.

### What Motion Should NOT Do

- No bounce or elastic on exit animations
- No celebration animations when tasks are completed (a quiet checkmark is enough)
- No skeleton loaders — the app is local-first; data is instant
- No loading spinners for local operations

## Interaction Patterns

### Keyboard Navigation

The keyboard is the primary interface. Every view must be fully navigable without a mouse. This is not a "nice to have" — it is the product.

Current shortcut grammar:

```
↑ / ↓           — navigate tasks
← / →           — switch views
Enter           — toggle done/undone
Backspace       — archive
Escape          — close palette / deselect
D W P E         — open palette pre-focused on chip
N E D C O       — project-level actions
Cmd+K           As— command palette
```

**The arrow key guarantee**: Navigation must always select something. Never leave the user in a state where arrow keys do nothing (see memory: navigation_always_select.md).

### Command Palette

The palette (`Cmd+K`) is the primary task creation surface. It should feel like a professional tool, not a search box. Chips (`@p`, `@w`, `@d`) are the input grammar — they create structure without requiring forms.

### Optimistic UI

All writes are immediate. Never show a pending state for local operations. The sync badge communicates background state — it should never block the user's flow.

### Focus Ring

Orange outline at 1px, no offset, inset. Never rounded. This is both an accessibility requirement and a brand statement.

## Component Patterns

### Task Row

```
[space-dot] [checkbox] [title ————————————] [project] [date]
36px height, px-3, gap-3
```

- Space dot: 8px square, space color, no border
- Checkbox: 16px square, 0.5px border, zero radius
- Title: flex-1, truncate, Geist 500 14px -0.02em
- Project (optional): emoji + italic name, text-muted
- Date: JetBrains Mono 10px, tabular-nums, red if late
- Late indicator: SVG triangle, no background change, no border change

**Focus state**: `bg-[#FF4F00]/5` + `box-shadow: 0 0 8px rgba(255,79,0,0.4)`. This is the laser focus. Nothing else changes — not border, not text color.

### Sidebar Navigation

⚠️ **Current issue**: `border-l-2` on active sidebar links is in the banned patterns (side-stripe border > 1px). This should be replaced with either:

- A background tint (`bg-accent/8`) with no border
- A leading dot or accent square
- Full `border-[0.5px] border-accent` (all sides, retina weight)

### Overlays (Palette / Modal)

- `backdrop-filter: blur(12px)` on the backdrop
- Zero border-radius on the panel itself
- `border-[0.5px] border-border` to define the panel edge
- `bg-surface` or `bg-bg` — never pure transparent

### Badges / Counts

Task counts in the nav: JetBrains Mono, 10px, accent color, tabular-nums. No background pill — the number floats next to the label. This is correct and should stay.

### Buttons

No rounded corners. Ever. Button hierarchy:

- **Primary**: `bg-accent text-white` — reserve for one action per view
- **Secondary**: `border-[0.5px] border-border-2 text-text`
- **Ghost**: `text-muted hover:text-text` — for nav-level actions
- **Destructive**: `text-red` ghost, only gains a border on hover

Never show more than one primary button in a view. If you find yourself needing two, one of them isn't primary.

### Empty States

Empty states should teach the interface, not apologize for the missing content.

```
Inbox:  "Nothing to triage. New items arrive here from the Courier."
Today:  "No tasks scheduled for today. Press Cmd+K to add one."
```

Keep it to one sentence. No illustration, no large icon. Muted text, center-aligned in the scroll area.

## Absolute Bans (Codified)

These are never acceptable in Sift regardless of context:

| Pattern                                | Why banned                                                    |
| -------------------------------------- | ------------------------------------------------------------- |
| `border-left/right > 1px` with a color | AI design tell. Replace with background tint or no indicator. |
| `background-clip: text` gradient       | Gradient text is decorative noise. Solid color only.          |
| `border-radius > 0`                    | Violates zero-radius constraint. Communicates imprecision.    |
| Glassmorphism (decorative blur)        | Blur is for backdrops only. Never on content surfaces.        |
| Skeleton loaders                       | App is local-first. Data is instant. Skeletons lie.           |
| Same padding everywhere                | Monotony destroys hierarchy. Vary spacing to create rhythm.   |
| More than one primary button per view  | Hierarchy collapses. Make a decision.                         |
| Hover tooltips on obvious controls     | Assumes the user needs hand-holding. Trust them.              |
| Loading states for local operations    | The db is synchronous. Never show a spinner for a write.      |

## Brand Moments

The few places where Sift expresses personality:

**The orange dot** in the topbar (Sift wordmark) — 8px square with laser glow. This is the logo. It's a task indicator, not a decorative element.

**The 2px orange bar** at the top of the sidebar — a quiet brand accent. The only non-zero-width border in the system, justified as a structural element.

**Orange selection highlight** (`::selection`) — every text selection shows orange. Brand presence even in utility.

**The late triangle** — a small SVG warning icon on overdue dates. Red, inline, no background. Clinical precision on failure states.

**The sync dot** — a 6px pulsing dot in the topbar. Communicates system state without modal interruption.

These moments are enough. Resist adding more.

## Anti-Patterns to Watch For

**"Just add a card"** — The temptation to wrap everything in a card container. Cards create hierarchy but also create padding, shadow, and visual weight that slows the eye. Sift's lists are flat by design.

**"Make it discoverable"** — Tooltip-on-hover for every action. Sift is for power users. Keyboard shortcuts are documented in the HintBar. Trust the user to read it once.

**"Break up the space"** — Adding dividers, separators, and section headers everywhere. Use whitespace and typographic contrast for separation. Visual dividers should be rare and meaningful.

**"Show the user their progress"** — Completion bars, streak counters, gamification. Sift is not a habit tracker. The reward is the clear list, not a number going up.

**"Soften it a bit"** — The temptation to add `rounded-sm` or `rounded` "just for this one element." Zero radius is non-negotiable. It is the product's visual signature.
